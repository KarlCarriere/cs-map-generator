# CS2 Road Importer Mod — Plan

- Status: Draft / Exploratory — not committed to build
- Date: 2026-05-19
- Companion project to `cs-mapgen`, not part of this repository

## TL;DR

Build a Cities: Skylines II mod that consumes the `roads.geojson` + `manifest.json` produced by `cs-mapgen` and programmatically lays roads onto the imported heightmap. The mod is a separate C# project (CS2's modding language is C# under Unity DOTS) — it does **not** live in this Python repo, but the GeoJSON/manifest schemas defined here are its input contract.

Effort estimate (gut-feel, not committed):

| Milestone | Effort | Confidence |
| --- | --- | --- |
| M0 — API spike: place one straight road segment from code in the editor | 1 weekend | Medium — depends on how mod-friendly the Net API is |
| M1 — MVP: dump every GeoJSON LineString as straight segments, one road type | 1–2 weekends after M0 | High once M0 is green |
| M2 — Real intersections + `highway_class → prefab` mapping | 2–3 weeks | Medium |
| M3 — Bezier curve fitting from polylines | 1–2 weeks | High (well-known algorithm) |
| M4 — Polished public release on Paradox Mods (docs, settings UI, error handling, version compat) | Months, ongoing | N/A |

The single biggest unknown is M0. **Do not commit to M1+ until M0 proves bulk network creation is feasible from a mod.**

## Goals

- Auto-place real-world OSM roads on a CS2 map that was generated by `cs-mapgen` from the same bbox.
- Preserve `highway_class` (motorway, primary, residential, …) by mapping each class to a chosen CS2 road prefab.
- Run in the **Map Editor** (not in-game gameplay), as a one-shot import action.
- Idempotent enough to re-run without duplicating roads (delete-and-replace, or detect existing imports).

## Non-Goals

- Decorations, buildings, zoning, water — out of scope. Roads only.
- In-gameplay imports. Editor-only is fine.
- Supporting arbitrary GeoJSON sources from third parties. Input contract is the `manifest.json` + `roads.geojson` pair produced by this generator. (Other sources can be added later if the schema is honoured.)
- Two-way sync (export from CS2 back to GeoJSON). One-way only.

## Background

What CS2 modding looks like today:

- Official modding toolchain ships with the game; mods are C# DLLs loaded into the Unity DOTS runtime.
- Distribution is via [Paradox Mods](https://mods.paradoxplaza.com/) and tools like Skyve.
- Harmony (HarmonyX) patching is supported; BepInEx is also used but the official path is preferred.
- Community mods that manipulate the Net (road network) system already exist and prove the API is reachable: Road Builder, Anarchy, Move It, Better Bulldozer. None of them do bulk OSM import.

What does **not** exist today (per a search in May 2026):

- No CS2 mod that ingests OSM / GeoJSON road data and builds roads. The CS1 ecosystem has it (OpenStreetMap Import, GeoSkylines, Cimtographer), but those are not portable — CS2's network model is a different ECS-based system.
- An overlay-only mod (MapImageLayer) exists but does not place actual roads.

## Architecture Overview

```
┌───────────────────────────┐         ┌──────────────────────────────────┐
│  cs-mapgen (this repo)    │         │  cs2-road-importer (new C# mod)  │
│                           │         │                                  │
│  bbox → heightmap.png     │         │  reads manifest.json + roads.geo │
│         worldmap.png      │ ──────► │  projects WGS84 → CS2 world      │
│         roads.geojson     │         │  maps highway_class → prefab     │
│         manifest.json     │         │  builds Net segments via API     │
└───────────────────────────┘         └──────────────────────────────────┘
```

The mod is a separate repo (suggested name: `cs2-road-importer` or `cs-mapgen-cs2-mod`). The contract between them is the `ExportManifest` schema already produced by this generator — specifically `bounds` (WGS84 bbox + CRS) and the path to `roads.geojson`.

### Why the manifest matters

The mod needs to know **the exact bbox the heightmap was generated from** so it can project lon/lat onto CS2 world coordinates. Without `manifest.json`, the mod would have to ask the user to retype the bbox, which is error-prone. The manifest is the source of truth.

## Technical Pieces

### 1. GeoJSON ingestion

- Parse `roads.geojson` (RFC 7946 `FeatureCollection`, LineStrings, properties: `highway_class`, `length_metres`).
- C# stdlib `System.Text.Json` works; if perf is an issue use `GeoJSON.Net` or roll a minimal reader.
- Validate against `manifest.json["artifacts"]` SHA-256 to catch user error (wrong roads file).

**Risk:** low. Pure parsing.

### 2. Coordinate projection (WGS84 → CS2 world space)

This is the **most subtle correctness issue.** The pipeline must be:

```
(lon, lat)  →  local metric projection (the one this generator used)  →  normalize to bbox  →  CS2 world coords  →  sample terrain Y from imported heightmap
```

The local metric projection is described in [ADR 0001](./adr/0001-pipeline-architecture.md) §7 — UTM zone derived from bbox centroid, equal-area fallback for very large extents. The mod **must use the same projection** the generator used, otherwise roads will drift relative to terrain features.

Two options:

- **Option A:** Bake the projection parameters (UTM zone, false easting/northing, or the proj4 string) into the manifest, so the mod can reproduce the projection exactly. Recommended.
- **Option B:** Mod runs its own UTM-zone heuristic from `bounds`. Brittle near zone boundaries.

CS2's playable area is 14336 m × 14336 m (per Paradox dev diaries; verify before relying on this). The generator centers the heightmap on the bbox centroid; the mod must do the same.

**Action item for this generator:** add `projection` to `ExportManifest` (proj4 string or EPSG code + zone). Currently `manifest.json` only has `crs_epsg: 4326` on `bounds`, which is the input CRS, not the working CRS.

**Risk:** medium. Math is well-understood, but mismatched conventions will silently produce roads-that-look-right-but-are-off-by-100m.

### 3. Road type mapping (`highway_class` → CS2 prefab)

OSM `highway` values used by this generator (see `PrepareRoadsStage`):

```
motorway, trunk, primary, secondary, tertiary,
residential, unclassified, service, …
```

CS2 ships a fixed catalogue of road prefabs (Highway, 6-lane Avenue, 4-lane Avenue, 2-lane Road, Alley, etc.). The mapping is opinionated and the user should be able to override it.

Suggested ship-default mapping:

| OSM `highway` | CS2 prefab |
| --- | --- |
| `motorway`, `motorway_link` | Highway |
| `trunk`, `primary` | 6-lane Avenue |
| `secondary` | 4-lane Avenue |
| `tertiary` | 4-lane Road |
| `residential`, `unclassified` | 2-lane Road |
| `service` | Alley |
| anything else | (skip) |

Implementation: a settings panel exposing a JSON config the user can edit. Round-trip in mod settings.

**Risk:** low — pure configuration.

### 4. **API spike (M0): can we create Net segments in bulk?**

This is the unknown that gates the entire project. The community Road Builder mod creates **one** road interactively. We need to call the same API in a loop, ideally without firing the editor's UI for each segment.

Open questions to answer in the spike:

- Is there a public `NetTool.Create(segment)` or equivalent in the modding API, or is the entire road-placement pipeline driven by user-input events?
- Can segments be created in a single ECS command-buffer frame, or does each segment require a tick to commit?
- What's the cost of placing N segments? For Québec City at residential resolution that's likely 5,000–20,000 segments.
- Does the editor's "Anarchy" mode need to be programmatically enabled to bypass placement-rules (e.g., slope limits) for terrain-intensive roads?

Concrete spike deliverable: a 50-line mod that, on hotkey, places three hardcoded straight road segments connecting four hardcoded world-space points. If that works, M1 is a straightforward extension.

**Risk:** high. If the API only accepts user-input-driven placement, the project pivots to "guided overlay + click-to-build" instead of full auto-import.

### 5. Intersection resolution

OSM LineStrings share coordinate tuples at intersections; CS2 has explicit `NetNode` entities at intersections. The importer must:

1. Build a coordinate → node-id map across all LineStrings.
2. Split LineStrings at every coordinate that appears in more than one feature.
3. Insert a CS2 NetNode at each shared coordinate, then create segments between them.

The generator already produces deterministic edge ordering (sorted by `(source, target)`), which helps. Whether the GeoJSON output preserves intersection-coordinate identity exactly needs verification — floating-point equality on lon/lat is fragile; a coordinate-snap tolerance (e.g., 0.1 m in projected coords) is probably required.

**Action item for this generator:** consider emitting a separate `intersections.geojson` with explicit Point features for shared nodes, so the mod doesn't have to re-derive topology from LineString geometry.

**Risk:** medium. Standard graph problem; the snap-tolerance choice has correctness implications.

### 6. Curve fitting (Bezier)

CS2 roads support cubic Bezier curves; OSM gives you polylines. Straight-segment-per-polyline-edge will produce ugly, high-node-count roads.

Options, easiest → best:

- **Step 1:** straight segments. Ship M1 with this.
- **Step 2:** Douglas-Peucker simplification of the polyline to reduce node count before placing straight segments.
- **Step 3:** Bezier fitting (Schneider 1990 algorithm). Each output Bezier covers many input vertices. Yields curvature that looks natural.

**Risk:** low — well-known algorithms, just engineering work.

### 7. Terrain conformance

CS2 roads modify terrain (cut/fill). Two failure modes:

- Importing thousands of roads triggers thousands of terrain edits → slow, possibly stalls the editor.
- Roads with steep underlying terrain will fail placement-rules unless Anarchy mode is on.

Mitigations:

- Place roads in batches; let the editor settle between batches.
- Enable Anarchy mode programmatically (if the API permits) for the duration of the import.
- Pre-filter: drop OSM segments whose underlying slope exceeds the CS2 limit for the chosen prefab.

**Risk:** medium. Performance unknowns until the spike runs.

### 8. UX

Minimal viable UI:

- New button in the Editor menu: "Import roads from cs-mapgen".
- File picker for `manifest.json` (the mod resolves `roads.geojson` from the manifest path).
- Settings panel for `highway_class → prefab` overrides.
- Progress bar (N of M segments placed).
- Idempotency: track imported segments via a custom ECS component so a second import deletes-then-rebuilds rather than duplicating.

## Open Questions

- **Q1:** Is bulk Net creation supported via the modding API, or is it user-input-driven only? (Gates everything. Answer via M0 spike.)
- **Q2:** What's the exact playable-area extent in CS2 world units, and is it stable across map sizes? (Affects projection math.)
- **Q3:** Does the heightmap-import flow produce world coordinates that exactly match `bounds` from the manifest, or is there an editor-side offset/scale? (See [ADR 0002](./adr/0002-cs2-export-format.md) Q1–Q4.)
- **Q4:** Can Anarchy mode be toggled programmatically by a mod, or does the user have to enable it manually first?
- **Q5:** What's the per-segment placement cost (ms)? Determines whether 10k-segment imports are feasible in one session.

## Milestones (phased)

### M0 — API spike

- Goal: prove `NetTool` (or equivalent) is callable from mod code for bulk creation.
- Deliverable: 50-line mod that places 3 hardcoded straight segments on hotkey.
- Exit criteria: segments visible in the editor, no exceptions in the log, repeatable.
- **Gate:** if this fails, scope changes radically. Possibly pivot to "visual overlay only".

### M1 — Straight-segment MVP

- Reads `manifest.json` from a file picker.
- Reads `roads.geojson`, projects coordinates, places every LineString as a chain of straight segments.
- One hardcoded road prefab for all roads. No intersection logic — overlapping segments are tolerated.
- Console-only progress / error reporting.

### M2 — Real intersections + class mapping

- Builds shared-node graph, places explicit CS2 NetNodes at intersections.
- `highway_class → prefab` table; default mapping in §3 above.
- Settings UI for overriding the mapping.

### M3 — Curve fitting

- Bezier fit per LineString (Schneider 1990 or simpler).
- Optional Douglas-Peucker pre-simplify.

### M4 — Polish + release

- Idempotent re-import.
- Anarchy-mode integration for steep terrain.
- Localized settings UI.
- Paradox Mods publication, screenshots, README, version compat matrix.
- Telemetry/logging discipline for bug reports.

## Changes Needed in `cs-mapgen` to Support the Mod

These are this-repo changes that make the mod easier to build. None are blocking, but doing them now means the mod can ship without revisiting the manifest format later:

1. **Add `projection` to `ExportManifest`** — proj4 string or `{ epsg: int, zone: int }`. The mod must reproduce the working CRS exactly. (See §2.) Bump `schema_version` to 2.
2. **Optional: emit `intersections.geojson`** — explicit Point features at shared LineString endpoints, so the mod doesn't snap-tolerance-guess topology. (See §5.)
3. **Document `highway_class` enum** — the exact set of values `PrepareRoadsStage` emits, so the mod's mapping table can be exhaustive. Add to `docs/ARCHITECTURE.md`.
4. **Stretch: emit `playable_extent_metres`** — the working-CRS bbox of the playable area (e.g., 14336 m × 14336 m for CS2) so the mod doesn't have to know engine-specific constants.
