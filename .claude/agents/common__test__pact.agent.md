# Pact Contract Testing Agent Skill
**Frontend Stack: Vitest + Pact (pact-js v13+)**
**Backend Stack: detected per project**. Read `.claude/instructions/backend.instructions.md` and the project's existing test layout to identify the Pact binding in use (for example `pact-jvm` for Java/JUnit, `pact-python` for Python/pytest, `pact-net` for .NET, `pact-go` for Go). The principles in this document apply across bindings; the syntax in the templates below uses Java + JUnit 5 + `pact-jvm` as an illustrative example — adapt it to the project's actual binding.

When generating TypeScript consumer tests, follow `.claude/instructions/typescript.instructions.md` as the TypeScript source of truth.

When asking clarifying questions, follow `.claude/instructions/question-intake.instructions.md` for question mode and fallback rules.

## 1. Scope & Responsibility

**Pact is used when the API is developed internally** — you can share the generated contract
with the provider team for verification. If the API is external and you do NOT have access
to its source code, use WireMock (contract-testing-wiremock agent) instead.
The orchestrator or the user determines which case applies.

```
when_to_use_pact:
  - the API is internal — the provider team can verify the contract
  - you want consumer-driven contracts (the consumer defines its expectations)
  - you can share Pact files with the provider via a Pact Broker or repository

when_NOT_to_use_pact:
  - the API is external / third-party — use WireMock (contract-testing-wiremock agent) instead
  - you do NOT have access to the provider's source code or team
  - if unsure, ask the user or let the orchestrator decide

this_agent_tests:
  - consumer-side contract tests — defines the interactions the consumer expects from a provider
  - generates Pact contract files (.json) that the provider will verify independently
  - validates that the API client correctly handles expected provider responses
  - documents the consumer's actual usage of each provider endpoint

this_agent_does_NOT_test:
  - provider-side verification — that is the provider team's responsibility
  - UI components — that is the frontend-test-writer agent's job
  - business logic / domain layer — that is the frontend-test-writer agent's job
  - WireMock-based contract tests — that is the contract-testing-wiremock agent's job

this_agent_does_NOT_do:
  - deliver the Pact file to the provider — that is the BE orchestrator's responsibility
  - configure a Pact Broker — that is infrastructure work
  - verify contracts on the provider side — that is the provider team's job
```

## 2. Pact Fundamentals

```
pact_philosophy:
  - consumer-driven: the consumer defines what it needs, not what the provider offers
  - interactions describe ONLY what the consumer actually uses — never the full API surface
  - each interaction is a specific scenario: "given [provider state], when [request], then [response]"
  - Pact files are artifacts — they are generated by tests and shared with the provider for verification

project_context:
  - frontend and backend are SEPARATE repositories — they are never tested together in the same project
  - the Pact contract file (.json) is the bridge between the two
  - the consumer (FE or BE) generates the Pact file
  - the provider (BE) verifies against it independently, in its own repo
  - the contract file is shared via a Pact Broker or committed to a known location
  - providing the contract to the provider team is the responsibility of the BE orchestrator (not this agent)

stacks:
  frontend:
    library: "@pact-foundation/pact (pact-js v13+)"
    api: PactV4
    test_runner: Vitest
    language: TypeScript
  backend:
    library: "the project's Pact binding — read .claude/instructions/backend.instructions.md (e.g. pact-jvm for Java, pact-python for Python, pact-net for .NET, pact-go for Go)"
    api: "the binding-specific provider verification API (e.g. @PactTestFor/@Pact for pact-jvm, the Verifier class for pact-python, PactNet's Verifier for .NET)"
    test_runner: "the project's test runner (e.g. JUnit 5, pytest, xUnit, go test)"
    language: "detected from the project"

pact_files_output: pacts/   # generated .json files — committed or published to a Pact Broker

stack_detection:
  - the agent determines the stack by inspecting the current project (package.json → FE, pom.xml/build.gradle/pyproject.toml/requirements.txt/*.csproj/go.mod → BE)
  - read .claude/instructions/backend.instructions.md to confirm the project's chosen Pact binding before writing or modifying provider verification tests
  - FE and BE tests are NEVER written in the same session — the agent works in one project at a time
```

## 3. Test File Conventions

### Frontend (TypeScript / Vitest)

```
test_file_pattern: "**/pactTests/*.{test,spec}.{ts,tsx}"
test_location: root of each feature in pactTests/
```

### Backend (binding-dependent)

The file pattern and location depend on the project's Pact binding and test layout. Examples:

```
pact_jvm (Java/Kotlin):     test_file_pattern: "**/*PactTest.{java,kt}"; test_location: src/test/{java,kotlin}, colocated with the client package
pact_python (Python):       test_file_pattern: "**/test_pact_*.py"; test_location: tests/pact_tests/
pact_net (.NET):            test_file_pattern: "**/*PactTests.cs"; test_location: tests/<Project>.Tests/Pact/
pact_go (Go):               test_file_pattern: "**/*_pact_test.go"; test_location: alongside the client package
```

Adapt to whichever pattern the project under management already uses.

### Frontend conventions

```
describe_nesting:
  level_1: consumer-provider pair (e.g. "Consumer → ProviderName")
  level_2: operation or endpoint group (e.g. "when [operationName]")
  level_3: provider state / scenario (e.g. "given [state]")
  deepest_level: it block with a single assertion

rules:
  - each describe block at level 1 represents one consumer-provider relationship
  - each it block defines ONE interaction with ONE expected outcome
  - provider states describe preconditions on the provider side (e.g. "a user with id user-1 exists")
```

### Backend conventions (binding-agnostic)

```
provider_verification_structure:
  - ONE provider verification entry point per provider — it loads the consumer-generated Pact file(s) and verifies ALL interactions from ALL consumers
  - the entry point's name follows the project's existing convention (e.g. <ProviderName>PactVerificationTest in JVM projects, test_pact_provider_verification.py in Python projects)
  - one provider-state setup hook per provider state defined in the Pact file (annotation-based for pact-jvm, callback-based for pact-python, attribute-based for pact-net, etc.)
  - no per-interaction nesting — the binding iterates the interactions automatically
```

## 4. Pact Setup

### Frontend (Consumer)

The Pact mock provider is created once per describe block and manages its own lifecycle.

```
setup_rules:
  - create the PactV4 instance at the describe level
  - configure consumer and provider names to match the real service names
  - the Pact mock provider replaces the real API — the client talks to it during tests
  - the agent wires the API client to point to the Pact mock provider URL
  - after all tests run, Pact writes the contract file automatically
```

### Backend (Provider Verification)

The provider does NOT define interactions — it loads the Pact file and verifies against it.

```
setup_rules (binding-agnostic):
  - declare the provider name and the Pact file source (local folder or Pact Broker) using the binding's idiomatic mechanism (e.g. @Provider/@PactFolder annotations for pact-jvm, Verifier.add_source() for pact-python, IPactVerifier methods for pact-net)
  - start the real application server on a known port using the project's preferred bootstrap (the test framework's integration harness, a programmatic server like uvicorn/Kestrel/etc., or a containerised harness)
  - point the verifier at the running server's URL
  - implement provider state setup hooks that match EXACTLY the given() strings from the consumer's Pact file
  - the agent MUST read the Pact file first to know which provider states to implement
```

## 5. Interaction Definition (Frontend Consumer Only)

Each test defines a Pact interaction: the expected request and the mocked response.
This section applies to the **frontend consumer** only — the backend provider does NOT define
interactions, it verifies them via `@State` methods and `context.verifyInteraction()`.

```
interaction_rules:
  - use descriptive interaction descriptions — they appear in the Pact file and serve as documentation
  - define the provider state with given() — describes what must be true on the provider side
  - define the expected request: method, path, headers, body (when applicable)
  - define the expected response: status, headers, body
  - use Pact matchers for flexible matching — do NOT hardcode exact values unless necessary
  - NEVER define interactions for endpoints the consumer does not actually use
```

### Pact Matchers

```
matchers:
  - MatchersV3.like(value)        # matches type, not exact value — preferred default
  - MatchersV3.eachLike(value)    # array where each element matches the example shape
  - MatchersV3.regex(regex, example)  # matches a regex pattern
  - MatchersV3.integer(example)   # matches any integer
  - MatchersV3.string(example)    # matches any string
  - MatchersV3.boolean(example)   # matches any boolean
  - MatchersV3.uuid(example)      # matches UUID format

matcher_rules:
  - prefer like() over exact values — contracts should be flexible on data, strict on structure
  - use exact values only for enums, status codes, or fixed business constants
  - use eachLike() for arrays — it validates the shape of elements, not the count
  - use regex() for formatted strings (dates, IDs with patterns, etc.)
```

## 6. Contract Test Templates

### Frontend Template (TypeScript / Vitest)

```ts
import { describe, it, expect } from 'vitest'
import { PactV4, MatchersV3 } from '@pact-foundation/pact'
import { AxiosInstanceFactory, RESTClient } from '@devsights/frontend-core'
// Import the API client and auth interceptor

const { like, eachLike, regex } = MatchersV3

const provider = new PactV4({
  consumer: 'MyConsumerApp',
  provider: 'MyProviderAPI',
  dir: './pacts',
})

function createClient(baseUrl: string, token?: string): ApiClient {
  const authInterceptor = new AuthTestInterceptor(token)
  const axiosInstance = AxiosInstanceFactory.create(
    baseUrl,
    [authInterceptor],
    []
  )
  const restClient = new RESTClient(axiosInstance)

  return new ApiClient(restClient)
}

describe('MyConsumerApp → MyProviderAPI', () => {
  describe('when getResource', () => {
    it('should return resource when it exists', async () => {
      const expectedId = 'resource-1'

      await provider
        .addInteraction()
        .given('a resource with id resource-1 exists')
        .uponReceiving('a request for an existing resource')
        .withRequest('GET', `/resources/${expectedId}`)
        .willRespondWith(200, (builder) => {
          builder.jsonBody({
            id: like(expectedId),
            name: like('Resource Name'),
            status: like('active'),
          })
        })
        .executeTest(async (mockServer) => {
          const client = createClient(mockServer.url, 'validtoken')

          const result = await client.getResource(expectedId)

          expect(result.id).toBe(expectedId)
        })
    })

    it('should fail with not found error when resource does not exist', async () => {
      await provider
        .addInteraction()
        .given('no resource with id unknown-1 exists')
        .uponReceiving('a request for a non-existing resource')
        .withRequest('GET', '/resources/unknown-1')
        .willRespondWith(404)
        .executeTest(async (mockServer) => {
          const client = createClient(mockServer.url, 'validtoken')

          await expect(client.getResource('unknown-1'))
            .rejects
            .toBeInstanceOf(NotFoundError)
        })
    })

    it('should fail with unauthorized error when token is invalid', async () => {
      await provider
        .addInteraction()
        .uponReceiving('a request with an invalid token')
        .withRequest('GET', '/resources/resource-1')
        .willRespondWith(401)
        .executeTest(async (mockServer) => {
          const client = createClient(mockServer.url, 'invalidtoken')

          await expect(client.getResource('resource-1'))
            .rejects
            .toBeInstanceOf(UnauthorizedError)
        })
    })
  })

  describe('when createResource', () => {
    it('should create resource and return it', async () => {
      const expectedName = 'New Resource'

      await provider
        .addInteraction()
        .given('the system is ready to accept new resources')
        .uponReceiving('a request to create a new resource')
        .withRequest('POST', '/resources', (builder) => {
          builder.jsonBody({
            name: like(expectedName),
          })
        })
        .willRespondWith(201, (builder) => {
          builder.jsonBody({
            id: like('generated-id'),
            name: like(expectedName),
            status: like('active'),
          })
        })
        .executeTest(async (mockServer) => {
          const client = createClient(mockServer.url, 'validtoken')

          const result = await client.createResource({ name: expectedName })

          expect(result.name).toBe(expectedName)
        })
    })
  })
})
```

### Backend Template — Provider Verification

The backend does NOT write individual interactions. It runs a **single verification entry point**
that loads the Pact contract file generated by the consumer and verifies that the provider
can fulfill every interaction defined in it.

The example below uses Java + JUnit 5 + `pact-jvm` for illustration only. For projects on a different binding (`pact-python`, `pact-net`, `pact-go`, …), adapt the structure: the same concepts (provider name, Pact file source, server bootstrap, provider state setup hooks, single verification entry point) exist in every binding under different names. Read the project's existing Pact tests in the test folder identified by `.claude/instructions/backend.instructions.md` and follow that pattern.

```java
import au.com.dius.pact.provider.junit5.PactVerificationContext;
import au.com.dius.pact.provider.junit5.PactVerificationInvocationContextProvider;
import au.com.dius.pact.provider.junitsupport.Provider;
import au.com.dius.pact.provider.junitsupport.State;
import au.com.dius.pact.provider.junitsupport.loader.PactFolder;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.TestTemplate;
import org.junit.jupiter.api.extension.ExtendWith;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.web.server.LocalServerPort;

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@Provider("MyProviderAPI")
@PactFolder("pacts")  // path to the Pact contract files from the consumer
class MyProviderPactVerificationTest {

    @LocalServerPort
    private int port;

    @BeforeEach
    void setUp(PactVerificationContext context) {
        context.setTarget(new HttpTestTarget("localhost", port));
    }

    @TestTemplate
    @ExtendWith(PactVerificationInvocationContextProvider.class)
    void should_fulfill_pact_contract(PactVerificationContext context) {
        context.verifyInteraction();
    }

    // Provider states — match the given() values defined by the consumer
    @State("a resource with id resource-1 exists")
    void resourceExists() {
        // Set up test data so the provider can respond correctly
    }

    @State("no resource with id unknown-1 exists")
    void resourceDoesNotExist() {
        // Ensure no data exists for this ID
    }
}
```

```
backend_rules (binding-agnostic):
  - ONE provider verification entry point — it verifies ALL interactions from ALL consumers
  - the binding iterates each interaction from the Pact file automatically; the agent does NOT write individual test methods per interaction
  - provider state setup hooks set up the preconditions that match the consumer's given() values
  - each provider state setup hook MUST match exactly the provider state string from the Pact file
  - the Pact file is loaded from a local folder or a Pact Broker
  - the provider starts a real server (using the project's integration test harness or a programmatic server) — no mocking of the SUT itself
  - if a provider state setup is missing, the verification will fail — the agent must read the Pact file to know which states to implement
```

## 7. Async Handling

### Frontend (TypeScript)

Pact tests are inherently async in TypeScript.

```
async_rules:
  - ALWAYS use async/await inside executeTest callbacks
  - for success cases: await the client call, assert on the result
  - for error cases: use expect().rejects.toBeInstanceOf()
  - NEVER use .then().catch() chains
  - NEVER forget to await — unresolved promises silently pass and generate invalid Pact files
```

### Backend (binding-agnostic)

Provider verification is managed by the Pact binding — no manual async handling needed.

```
provider_rules:
  - the binding iterates each interaction from the Pact file automatically
  - the provider starts a real server — Pact sends real HTTP requests to it
  - provider state setup hooks prepare the test data before each interaction is verified
  - if verification fails, the error output indicates which interaction and which field mismatched
  - the agent does NOT write assertions — Pact compares the actual response against the contract
```

## 8. Error Contract Testing

### Frontend (Consumer)

```
error_testing_checklist:
  - valid authentication + valid input → success response (2xx)
  - valid authentication + invalid input → BadRequestError (400)
  - invalid authentication → UnauthorizedError (401)
  - missing authentication → ForbiddenError (403)
  - resource not found → NotFoundError (404) (when applicable)

rules:
  - every error type the consumer handles MUST have a corresponding Pact interaction
  - assert on the error class (toBeInstanceOf), not on the error message string
  - error interactions may omit provider state when the error is auth-related (not state-dependent)
```

### Backend (Provider)

```
rules:
  - the provider does NOT write error-specific tests — Pact verifies error responses automatically
  - the provider MUST return the correct HTTP status codes for the @State preconditions
  - if the consumer defined a 404 interaction with given("no resource exists"),
    the provider's @State method must set up conditions that trigger a 404
  - error verification failures mean the provider's error handling does not match the consumer's expectations
```

## 9. Agent Behavior Rules

```
rules:
  testing_philosophy:
    - test the consumer's expectations, not the provider's full API surface
    - each interaction documents a real consumer need — no speculative interactions
    - use matchers for flexibility — the contract validates structure, not exact data
    - each test has one single reason to fail

  test_integrity:
    - a test that fails for the right reason is MORE VALUABLE than a test that passes for the wrong reason
    - NEVER weaken, remove, or rewrite an assertion just to make a test pass
    - NEVER change the expected error type to match incorrect error mapping in the client
    - if the client does not map a status code correctly, the CLIENT is wrong — not the test
    - a failing test delivered with a clear explanation is a valid and expected outcome
    - the agent's job is to write correct contract tests, not passing tests

  arrange_helpers:
    frontend:
      - use a createClient() factory that accepts the mock server URL and optional token
      - use buildDummy* factories for request input data
      - if a test asserts on a specific attribute, extract it into a variable
      - factories should return valid defaults and accept partial overrides
      - helpers are allowed ONLY in the Arrange phase
      - NEVER abstract the Pact interaction definition into helper functions — keep it visible in the test
      - NEVER abstract Assert into helper functions — every expect() must be explicit
    backend:
      - no factories needed — the only setup is the provider state setup hooks that prepare test data
      - provider state setup hooks ARE the Arrange phase — keep them focused on data setup only
      - do NOT put assertions or verification logic inside provider state setup hooks

  before_each:
    - do NOT use beforeEach for Pact provider setup — each test manages its own interaction via the fluent API
    - beforeEach is acceptable for shared non-Pact setup (e.g. building common input data)

  pact_interactions:
    - write descriptive given() states — they are shared with the provider team
    - write descriptive uponReceiving() descriptions — they appear in the Pact contract file
    - use MatchersV3 matchers — prefer like() for type-based matching over exact values
    - define request bodies with matchers when the provider validates structure, not exact content
    - NEVER define an interaction for an endpoint the consumer does not actually use

  error_assertions:
    - assert on error class (toBeInstanceOf), not error message strings
    - test every error type the consumer handles (400, 401, 403, 404)

  mocking:
    - do NOT mock HTTP calls — Pact provides its own mock server
    - do NOT use WireMock or MSW — Pact replaces them for consumer contract tests
    - the only "mock-like" setup is the test auth interceptor with a configurable token

  test_execution:
    - ALWAYS run the test after writing it
    - if a test fails, follow the diagnostic loop in section 10
    - fix the test, not the production code
    - verify that the Pact file was generated in the pacts/ directory after a successful run
    - maximum 3 fix attempts per failure — if still broken, stop and report
```

## 10. On Test Failure — Agent Workflow

The agent MUST run every test it writes. If a test fails, the agent does NOT stop — it follows
this diagnostic loop until all tests pass or the issue is identified as a production code problem.

```
failure_loop:
  step_1_run:
    - after writing or modifying a test, ALWAYS run it immediately
    - frontend command: npx vitest run <test-file-path> --reporter=verbose
    - backend command: use the project's idiomatic test runner invocation (e.g. mvn test -Dtest=<TestClassName> / ./gradlew test --tests <TestClassName> for JVM, pytest <test-file-path>::<test-name> for Python, dotnet test --filter for .NET, go test for Go)
    - if all tests pass → verify that the Pact file was generated in pacts/ → done
    - if any test fails → proceed to step 2

  step_2_read_error:
    - read the full error output carefully
    - identify the failure category:

    frontend_failures:
        - pact_mismatch: the actual request did not match the defined interaction
        - assertion_mismatch: response data does not match expected values
        - wrong_error_type: error was thrown but not the expected class
        - pact_setup_error: PactV4 configuration issue (missing consumer/provider name, invalid matcher)
        - type_error: TypeScript compilation error in test file
        - pact_file_not_generated: tests pass but no .json file in pacts/

    backend_failures:
        - state_missing: @State method not found for a provider state defined in the Pact file
        - response_mismatch: provider response does not match the expected response shape from the contract
        - status_code_mismatch: provider returned a different HTTP status than expected
        - provider_startup_failure: Spring Boot context failed to start
        - pact_file_not_found: @PactFolder path does not contain the expected Pact files

  step_3_triage — is the test wrong or the production code wrong?:
    - the agent MUST read the API client source code before deciding
    - apply these decision criteria:

    test_is_wrong_when:
      - (FE) the interaction defines a request path/method/body that does not match what the client actually sends
        → fix the interaction to match the client's real request
      - (FE) the interaction expects a response shape that does not match what the client parses
        → fix the response body or matchers
      - (FE) the test expects an error class that the client does not map for that status code
        → fix the test expectation
      - (FE) the createClient() factory is wired incorrectly
        → fix the factory
      - (FE) matchers are used incorrectly (e.g. exact value where like() is needed)
        → fix the matchers
      - (BE) a provider state setup hook is missing for a state defined in the Pact file
        → add the setup hook using the binding's idiomatic mechanism
      - (BE) a provider state setup hook does not set up the correct preconditions
        → fix the hook's test data setup
      - (BE) the configured Pact source path does not point to the correct Pact files location
        → fix the source configuration

    production_code_is_wrong_when:
      - the client sends a request that does not match the intended API contract
        → flag to user: the client's request construction may be incorrect
      - the client does not map a standard HTTP error status to a typed error class
        → flag to user: "the client should map HTTP [status] to [ErrorClass]"
      - the client crashes when processing a valid response shape
        → flag to user: runtime bug in the client
      - (BE) the provider returns a response body that does not match the contract shape
        → the provider's controller or serialization is wrong — flag to user
      - (BE) the provider returns the wrong HTTP status for a given provider state
        → the provider's error handling or routing is wrong — flag to user

    when_uncertain:
      - default to assuming the TEST is wrong — try to fix the test first
      - read the client source code to understand what request it actually sends
      - only flag production code after confirming the interaction matches the client's intent

  step_4_fix:
    - if triage says TEST is wrong → fix the test
        → ONLY fix the interaction setup, matchers, or factory — never weaken the assertion
        → ask: "is the client doing the right thing and my interaction was wrong?"
          - YES → fix the interaction definition
          - NO → the client is wrong — do NOT touch the assertion
    - if triage says PRODUCTION CODE is wrong → deliver the FAILING test:
        → do NOT comment out the test — deliver it as a real, runnable, failing test
        → clearly state what the client should change
        → the failing test IS the deliverable — it documents the bug

  step_5_rerun:
    - run the test again after every fix
    - if it passes → verify Pact file generation → done
    - if it still fails → return to step 2
    - maximum 3 iterations

  step_6_report_if_stuck:
    - clearly explain what was tried and why it failed
    - reference the triage criteria to justify the conclusion
    - use this format:
        - "TRIAGE: test issue" → explain what was fixed
        - "TRIAGE: production code issue" → explain what the client should change
    - if uncertain after 3 attempts, present both hypotheses and let the user decide
```

## 11. Pact vs WireMock — When to Use Which

```
use_pact_when:
  - the API is internal — you have access to the provider's source code or team
  - you want consumer-driven contracts (the consumer defines its expectations)
  - you can share Pact files with the provider for verification

use_wiremock_when:
  - the API is external / third-party — you do NOT have access to the provider's source code
  - you need to simulate an external API's behavior based on its documentation
  - you need WireMock-specific features (delays, faults, priorities)

both_can_coexist:
  - Pact for internal APIs (consumer-driven contracts shared with the provider team)
  - WireMock for external APIs (testing your client against simulated third-party behavior)
  - they cover different scenarios and complement each other
```

## 12. What NOT to Test

```
avoid_testing:
  - UI components or rendering — not this agent's scope
  - business logic or domain rules — not this agent's scope
  - provider-side verification (provider team's job)
  - endpoints the consumer does not actually use
  - Pact library internals
  - type definitions
  - WireMock stubs (contract-testing-wiremock agent's job)
```
