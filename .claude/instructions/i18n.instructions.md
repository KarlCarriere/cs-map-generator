# Internationalisation (i18n) Instructions

This file is the source of truth for **framework-agnostic** i18n principles across frontend agents.

The concrete i18n library (`react-i18next`, `vue-i18n`, `svelte-i18n`, `@angular/localize`, `next-intl`, …) is owned by the project's frontend coder agent (`.claude/agents/frontend__coder__<framework>.agent.md`) and the project's frontend stack source-of-truth (`.claude/instructions/frontend.instructions.md` when adopted). Always check current docs via `mcp__context7__get-library-docs` before adopting a new API surface.

## Strings And Translation Calls

- Never hardcode user-visible strings — every string must go through the i18n library's translation function (typically `t('…')` or its framework-equivalent).
- Use the library's element-aware component (`<Trans>`, `<i18n-t>`, …) for strings that contain inline markup or interpolated framework elements.
- Use dot-notated access for nested keys (e.g. `t('header.title')`, `t('header.descriptions.philosophy')`).
- Use named interpolation variables when the library supports them (e.g. `t('key', { name: user.name })`) — never positional interpolation when a named alternative exists.
- Do not store translated strings in component state — call the translation function at render time so locale changes are reactive.
- Never pass translated strings as props across component boundaries — pass the key and translate at the leaf component.

## Translation Key Design

- Organise translation keys by feature or component namespace using nested JSON objects (e.g. `header.title`, `header.descriptions.philosophy`).
- Prefer nested translation structures for readability and grouping by intent.
- Translation keys must be descriptive and intention-revealing — never use positional names like `text1` or `label2`.
- Do not construct translation keys dynamically using string concatenation — keys must be statically analysable so tooling can verify coverage and detect dead keys.

## Locale Files

- Keep all translation files in a dedicated `locales/` directory (e.g. `locales/en.json`, `locales/fr.json`).
- English (`en`) is the canonical source locale unless the project documents otherwise — all other locales derive from it.
- When adding a new string, add the key to all locale files simultaneously — leave non-English values empty rather than omitting the key, so missing translations stay visible to tooling.

## Testing And Mocking

- Canonical i18n mock file path: `src/mocks/i18n.ts` (or the project's equivalent). The exact location and contents come from the project's frontend instructions and bootstrap templates.
- The canonical mock uses **key passthrough** behaviour — the translation function returns the key it was given (e.g. `t('some.key')` returns `'some.key'`). This makes tests locale-agnostic and resistant to copywriting changes.
- Frontend tests should import or register this mock instead of hardcoding translated strings in assertions. Tests query and assert against the **translation key**, not against the rendered locale-specific copy.
- The framework-specific shape of the mock (which exports to stub, which provider to wrap, which composable to override) lives with the project's frontend coder agent and bootstrap templates.
